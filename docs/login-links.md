# Вход по ссылкам из Telegram и email

## Назначение

Зафиксировать единое поведение внешних ссылок и убрать неявные "автовходы" в обход Keycloak.

## Главное правило

- Ни Telegram-ссылка, ни email-ссылка не создают web-сессию напрямую.
- Пользователь всегда проходит стандартный вход через Keycloak.
- После входа пользователь возвращается на нужную страницу через параметр `next`.

## Текущее поведение

### Telegram

- Ссылки из Telegram ведут либо на регистрацию (`/api/v1/auth/oidc/register?...`), либо на `/login?next=...`.
- Маршрут `/api/v1/tg/auth` больше не логинит пользователя, а только переводит на `/login`.
- Страница `/auth/tg/login` показывает, что direct-вход из Telegram отключен, и предлагает обычный вход.
- Endpoint `/api/v1/auth/tg/exchange` возвращает запрет (`Forbidden`).

### Email

- Ссылки из email-уведомлений ведут на `/login?next=...`.
- Ссылка верификации email (`/api/v1/auth/verify-email`) только подтверждает email и не создает сессию.
- Magic-login по письму не используется.

## Для контрагента, созданного вручную

Если контрагент, созданный вручную, добавлен сотрудником с правами создания пользователей (например, администратором или экономистом):

1. Он все равно входит через обычный `/login`.
2. Для первого входа задает пароль через `Forgot password` на стороне Keycloak.
3. После успешного входа backend выполняет привязку к локальному пользователю.

## Что это дает

- Единый, предсказуемый и безопасный вход.
- Нет обхода Keycloak через "одноразовые" deep-link логины.
- Проще сопровождать аудит и политику безопасности.
