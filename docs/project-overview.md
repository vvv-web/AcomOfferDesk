## Назначение

`AcomOfferDesk` - это продуктовая система для работы с заявками и офферами между внутренними сотрудниками и контрагентами.

На верхнем уровне продукт решает четыре задачи:

1. Дать сотрудникам единый интерфейс для создания и сопровождения заявок.
2. Дать контрагентам контролируемый доступ к участию в заявках и подаче офферов.
3. Обеспечить прозрачную коммуникацию по офферу через рабочее пространство и чат.
4. Централизовать идентификацию, права доступа, уведомления и хранение файлов.

## Ключевые пользовательские роли

Роли хранятся в `users.id_role`:

- `superadmin`
- `admin`
- `contractor`
- `project_manager`
- `lead_economist`
- `economist`
- `operator`

Статус пользователя хранится в `users.status`:

- `review`
- `active`
- `inactive`
- `blacklist`

Практически вся бизнес-функциональность доступна только пользователям со статусом `active`.

## Основные сущности

### Заявка

Заявка создается сотрудником и описывает предмет запроса.

Обычно заявка содержит:

- описание;
- дедлайн;
- суммы;
- вложения;
- владельца;
- связанные офферы;
- служебную статистику по офферам и сообщениям.

### Оффер

Оффер - это ответ контрагента на заявку.

Оффер связан с:

- конкретной заявкой;
- конкретным контрагентом;
- суммой предложения;
- файлами;
- статусом;
- чатом внутри рабочего пространства оффера.

### Пользователь

Система различает:

- сотрудников;
- контрагентов, зарегистрированных по invite-ссылке;
- контрагентов, созданных вручную сотрудником;
- legacy-пользователей Telegram-потока.

### Файлы

Файлы используются в трех основных контекстах:

- вложения заявки;
- вложения оффера;
- вложения сообщений в чате.

Хранение файлов вынесено в S3-совместимое хранилище `MinIO`, а доступ к ним контролируется backend-правами.

## Ключевые бизнес-сценарии

### 1. Создание заявки сотрудником

Типовой поток:

1. Сотрудник проходит аутентификацию через Keycloak.
2. Во frontend открывается форма создания заявки.
3. SPA отправляет данные и файлы в backend.
4. Backend валидирует данные, сохраняет заявку, прикрепляет файлы и при необходимости запускает email-уведомления.

### 2. Участие контрагента в заявке

Контрагент может попасть в систему двумя основными способами:

- через email invite и самостоятельную регистрацию;
- через ручное создание сотрудником.

После активации контрагент:

- видит доступные заявки;
- открывает карточку заявки в contractor view;
- создает оффер;
- загружает файлы;
- участвует в чате по офферу.

### 3. Работа сотрудников с офферами

Сотрудники:

- просматривают список заявок;
- анализируют офферы по заявке;
- сравнивают предложения;
- общаются с контрагентом в рабочем пространстве оффера;
- принимают или отклоняют офферы;
- управляют владельцами, статусами и уведомлениями.

### 4. Управление пользователями

В системе есть административные сценарии:

- создание сотрудника;
- создание ручного контрагента;
- смена статуса пользователя;
- смена роли;
- переназначение менеджера;
- управление периодами недоступности.

## Как устроен продукт на уровне модулей

### `web`

Frontend-приложение на React.

Отвечает за:

- маршрутизацию;
- формы и экраны;
- работу с API;
- клиентскую авторизацию;
- отображение рабочих разделов по ролям;
- WebSocket-клиент для чата.

### `backend`

Основной API и бизнес-ядро системы.

Отвечает за:

- HTTP API;
- авторизацию и синхронизацию идентичности;
- бизнес-правила;
- работу с БД;
- управление файлами;
- генерацию ссылок регистрации и логина;
- чат и realtime-события;
- публикацию уведомлений.

### `notifications_worker`

Фоновый процесс, который обрабатывает сообщения из очереди и отправляет email-уведомления.

### `infra/keycloak`

Набор файлов для:

- bootstrap Keycloak;
- импорта realm;
- кастомной темы.

### `tg_bot`

Legacy-модуль для Telegram-сценариев. В основном runtime сейчас отключен по умолчанию, но код и документация сохранены для совместимости и возможного rollback.

## Что важно понимать новому разработчику

### 1. База данных внешняя

Основная рабочая PostgreSQL схема не живет в этом репозитории. Проект подключается к уже поднятой БД из отдельного репозитория `order_database`.

### 2. Gateway - это единая точка входа

Почти весь локальный runtime строится вокруг `gateway`, который:

- отдает SPA;
- проксирует `/api/*` в backend;
- проксирует `/iam/*` в Keycloak.

### 3. Авторизация не локальная, а через Keycloak

Даже если пользователь создается из админки, рабочий вход в web-интерфейс все равно завязан на Keycloak/OIDC.

### 4. В проекте много логики на пересечении ролей, статусов и permissions

При любом изменении, связанном с доступом, нужно смотреть одновременно:

- роль пользователя;
- статус пользователя;
- permissions;
- policy-слой;
- action flags, которые backend возвращает для UI.
