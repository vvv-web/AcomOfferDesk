# Автоматическая привязка Keycloak

## Назначение

Этот документ описывает, как в AcomOfferDesk выполняется первый вход через Keycloak для уже существующего локального пользователя без каких-либо изменений схемы БД.

Любая успешная привязка заканчивается созданием обычной записи в `user_auth_accounts`:

- `provider = 'keycloak'`
- `external_subject_id = <Keycloak sub>`

После первой успешной привязки все последующие входы идут уже через сохранённую связку.

## Основные правила

### Суперадминистратор

- Bootstrap-пользователь `superadmin` сохраняет существующее special-case поведение.
- При первом входе через Keycloak выполняется bootstrap binding к локальному пользователю `superadmin`.
- Доступ в систему открывается сразу.

### Режим `dev`

Этот режим используется для локальной разработки и быстрого тестирования, когда неудобно создавать уникальный реальный email для каждого тестового пользователя.

Flow:

1. Администратор создаёт локального пользователя с конкретным `username`.
2. Backend сразу создаёт или обновляет одноимённого пользователя в Keycloak.
3. Пользователь нажимает обычную кнопку `Войти через Keycloak`.
4. Backend получает claims от Keycloak.
5. Если в `user_auth_accounts` ещё нет привязки и включён dev auto-link, backend проверяет:
   - `preferred_username == users.id`
6. Если usernames совпадают строго, backend создаёт Keycloak-привязку и завершает вход.

Примечания:

- Локальный email у пользователя всё равно хранится, но в `dev` он не используется как ключ привязки.
- В `dev` Keycloak должен работать с `KEYCLOAK_VERIFY_EMAIL=false`, чтобы первый вход и автопривязка staff-пользователей не блокировались шагом подтверждения email.
- В `dev` пользователю не нужен отдельный шаг `Register` в Keycloak: аккаунт создаётся системой при создании локального пользователя.
- Этот режим запрещён в production и блокируется backend-конфигом.

### Режим `prod` для пользователей, созданных администратором

Этот режим используется для сотрудников и для контрагентов, созданных вручную.

Важно:

- email, который администратор вводит при создании пользователя, считается только локальным справочным значением
- такой email не считается верифицированным сам по себе
- фактическая верификация происходит только в момент первого входа через Keycloak, когда backend получает `email_verified=true`

Flow:

1. Администратор создаёт локального пользователя с конкретным `username` и конкретным локальным email.
2. Backend сразу создаёт или обновляет одноимённого пользователя в Keycloak.
3. Для сотрудников система задаёт Keycloak-пароль из формы создания пользователя.
4. Для вручную созданных контрагентов отдельный шаг `Register` не нужен: Keycloak-аккаунт уже существует, и первичная установка пароля может идти через обычный `Forgot Password` на форме Keycloak.
5. Пользователь нажимает обычную кнопку `Войти через Keycloak`.
6. После успешной аутентификации в Keycloak backend получает claims.
7. Если в `user_auth_accounts` ещё нет привязки и включён prod auto-link, backend проверяет:
   - `email_verified == true`
   - email из Keycloak однозначно соответствует одному локальному пользователю
   - текущий Keycloak subject ещё не привязан к другому пользователю
   - найденный локальный пользователь ещё не привязан к другому Keycloak subject
8. Если все проверки проходят, backend:
   - создаёт Keycloak-привязку
   - сохраняет email-канал пользователя как подтверждённый в `user_contact_channels`
   - завершает вход

Источники локального email для поиска:

- `profiles.mail`
- `company_contacts.mail`

Такой поиск покрывает и staff-пользователей, и вручную созданных контрагентов без добавления новых таблиц или полей.

Важно:

- для пользователей, созданных администратором, больше не требуется вручную создавать пользователя в админке Keycloak
- для таких пользователей больше не нужен отдельный шаг `Register` на форме Keycloak
- если в локальной системе один и тот же email указан у нескольких пользователей, production auto-link не сможет безопасно привязать вход без исправления данных
- для уже существующих локальных пользователей после выката такой логики нужен однократный sync Keycloak-аккаунтов

### Контрагенты, регистрирующиеся самостоятельно

Этот сценарий остаётся отдельным и концептуально не меняется:

1. Контрагент начинает регистрацию через Keycloak.
2. Keycloak подтверждает email контрагента.
3. Backend создаёт нового локального contractor-пользователя со статусом `review`.
4. Контрагент заполняет локальные бизнес-данные.
5. Доступ к бизнес-функциям открывается только после одобрения внутри AcomOfferDesk.

Этот flow не зависит от правил dev/prod auto-link для заранее созданных пользователей.

## Конфигурация

Основные флаги backend в `backend/.env`:

- `APP_ENV`
- `KEYCLOAK_VERIFY_EMAIL`
- `KEYCLOAK_DEV_AUTO_LINK_BY_USERNAME_ENABLED`
- `KEYCLOAK_PROD_AUTO_LINK_BY_VERIFIED_EMAIL_ENABLED`
- `KEYCLOAK_ADMIN_REALM`
- `KEYCLOAK_ADMIN_CLIENT_ID`

`KEYCLOAK_VERIFY_EMAIL` управляет именно поведением realm в Keycloak:

- в `development` должен быть `false`, чтобы не требовать подтверждение email до dev auto-link по `username == users.id`
- в `production` должен быть `true`, чтобы первый вход пользователей, созданных администратором, проходил через подтверждённый email

Backend использует Keycloak Admin API для автоматического создания Keycloak-пользователей при создании локальных пользователей. Для этого используются уже существующие переменные:

- `KC_BOOTSTRAP_ADMIN_USERNAME`
- `KC_BOOTSTRAP_ADMIN_PASSWORD`

Для сценариев, где Keycloak должен отправлять письмо подтверждения email, SMTP должен быть настроен именно в Keycloak realm.

В проекте это делается автоматически через `infra/keycloak/bootstrap.sh`: Keycloak использует SMTP-параметры из `backend/.env`.

Базовые переменные:

- `EMAIL_ADDRESS`
- `EMAIL_APP_PASSWORD`
- `EMAIL_FROM_NAME`
- `SMTP_HOST`
- `SMTP_PORT`

При необходимости можно задать отдельные override-переменные для Keycloak:

- `KEYCLOAK_SMTP_USERNAME`
- `KEYCLOAK_SMTP_PASSWORD`
- `KEYCLOAK_SMTP_FROM`
- `KEYCLOAK_SMTP_REPLY_TO`
- `KEYCLOAK_SMTP_FROM_DISPLAY_NAME`
- `KEYCLOAK_SMTP_SSL`
- `KEYCLOAK_SMTP_STARTTLS`

Рекомендуемые значения:

### Локальная разработка

```env
APP_ENV=development
KEYCLOAK_VERIFY_EMAIL=false
KEYCLOAK_DEV_AUTO_LINK_BY_USERNAME_ENABLED=true
KEYCLOAK_PROD_AUTO_LINK_BY_VERIFIED_EMAIL_ENABLED=false
```

### Production

```env
APP_ENV=production
KEYCLOAK_VERIFY_EMAIL=true
KEYCLOAK_DEV_AUTO_LINK_BY_USERNAME_ENABLED=false
KEYCLOAK_PROD_AUTO_LINK_BY_VERIFIED_EMAIL_ENABLED=true
```

Ограничения безопасности, которые проверяются backend:

- username auto-link и verified-email auto-link нельзя включать одновременно
- dev username auto-link нельзя включать при `APP_ENV=production`

## Ожидания от admin UI

Для пользователей, которые создаются не как контрагенты, форма создания в админке теперь сохраняет локальный email в `profiles.mail`.

Это обязательно для того, чтобы production auto-link по подтверждённому email действительно работал для пользователей, создаваемых через существующий web UI.

При этом нужно учитывать:

- email, введённый администратором, не создаёт подтверждённый контактный канал автоматически
- до первого успешного входа через Keycloak этот email остаётся неверифицированным
- подтверждённый email появляется только после первого успешного входа через Keycloak, если провайдер вернул `email_verified=true`

## Когда auto-link должен завершаться ошибкой

Автоматическая привязка намеренно блокируется в следующих случаях:

- по текущему правилу `dev` или `prod` не найден локальный пользователь
- в `prod` email из Keycloak не подтверждён
- один и тот же verified email соответствует нескольким локальным пользователям
- текущий Keycloak subject уже привязан к другому пользователю
- локальный пользователь уже привязан к другому Keycloak-аккаунту

В этих случаях вход не завершается, а на callback-странице показывается понятная ошибка.
